## 0. 概要

| 項目 | 内容 |
| --- | --- |
| タイトル | ノード・ドミニオン (Node Dominion) |
| ジャンル | 対称型・自律AIネットワーク陣取りRTS（非同期・対称型） |
| プレイ人数 | 1 vs 1（自前AI 対 NPC / 自前AI 対 自前AI） |
| 競うスキル | 操作技術ではなく **ネットワークトポロジーの設計力** と **AIへの作戦指示（IF-THENポリシー）の論理性** |
| テーマ | 抽象ネットワーク陣取り。サイバー攻撃の味付けを撤去し、本拠地・ノード・経路の取り合いに純化 |
| 勝利条件 | **敵本拠地を占領** で即時勝利。時間切れ時は **占領ノード数が多い側** の勝ち |
| マッチ構成 | 「ラウンド」= ①ビルドフェーズ → ②戦闘フェーズ の1セット。**Best of 3〜5** のラウンド制 |

### 0.1 プレイの基本ループ

```
┌──────────────────────────────────────────────────────────┐
│  マッチ開始（Best of 3〜5）                                  │
│                                                            │
│   ┌─ ラウンド N ──────────────────────────────────────┐    │
│   │                                                    │    │
│   │  ① ビルドフェーズ（操作あり / 制限時間あり）          │    │
│   │     - 初期ユニット配置・前線の組み立て               │    │
│   │     - AI作戦ポリシー（IF-THEN）の編集               │    │
│   │                                                    │    │
│   │  ② 戦闘フェーズ（操作不可 / マップ規模で制限tick）    │    │
│   │     - 両陣営AIが自律で進軍・占領・防衛               │    │
│   │     - サーバーが状態・判定・リソースを権威的に計算     │    │
│   │                                                    │    │
│   │  → 敵本拠地占領 or 時間切れ判定 → ラウンド勝敗確定     │    │
│   └────────────────────────────────────────────────────┘    │
│                                                            │
│   先に過半数ラウンドを取った側がマッチ勝利                    │
└──────────────────────────────────────────────────────────┘
```

プレイヤーが直接ユニットを動かす場面は **一切ない**。戦闘フェーズでは設計済みのトポロジー（前線配置）と作戦ポリシーだけが勝敗を決める。プレイヤーの「思考」がそのまま戦力になるゲームである。

> 本リポジトリには本設計を実証する **動くデモプレイ**（`demo/`）を同梱している。ブラウザで自律戦闘を観戦でき、**自前AI 対 NPC** の対戦を試せる。

### 0.2 マップ

盤面はノード（拠点）とエッジ（経路）からなるグラフ。両端に各プレイヤーの **本拠地** が置かれ、間を **中間ノード** が埋める。マップ規模で中間ノード数が変わる。

| 規模 | 中間ノード | 総ノード | おおよそのレーン | ラウンド制限 |
| --- | --- | --- | --- | --- |
| 小 (S) | 5 | 7（本拠地2＋中間5） | 2 | 120 tick |
| 中 (M) | 11 | 13（本拠地2＋中間11） | 3 | 180 tick |
| 大 (L) | 19 | 21（本拠地2＋中間19） | 3〜4 | 240 tick |

> 本作の時間単位は **1 tick = 1 ゲーム秒** とする（戦闘フェーズは離散tickで進行）。

#### ノード種別

| 種別 | 役割 | 初期防御 | 備考 |
| --- | --- | --- | --- |
| 本拠地 (home) | 防衛対象。占領されたら即敗北 | 要塞化Lv1相当 | SP収入 +5/tick |
| 補給ノード (supply) | 占領するとSP収入 | なし | SP収入 +3/tick/占領 |
| 要塞ノード (fortress) | 高防御のチョークポイント | 要塞化Lv2相当 | 攻めるには突撃(攻城)が要る |
| 通常ノード (normal) | 接続・前進拠点 | なし | 中立時は少数の守備が居る |

ノードの所有状態は {中立 / 自軍A / 敵軍B}。中立ノードは小規模な守備を持ち、これを排除して自軍ユニットが進入すると占領できる。

## 1. ゲームバランス

### 1.1 リソース二層制（陣取り版）

| リソース | 略称 | 型 | 入手 | 用途 | 性格 |
| --- | --- | --- | --- | --- | --- |
| Command Points | **CP** | フロー型（毎tick回復・上限あり） | 時間経過で自動回復 | AIの **行動コスト**（進軍・攻撃・生産・強化の命令） | 「今このtickに何手打てるか」 |
| Supply Points | **SP** | ストック型（貯蓄） | **占領ノードからの収入** | ユニット **生産・増援・拠点強化** | 「どれだけ盤面に投資できるか」 |

陣取りの核心はここにある。**SPは占領ノードから生まれる** ため、territory を広げるほど収入が増え、収入が増えるほど戦力を再生産できる（正のフィードバック）。これを一方的にしないための仕組みが 1.5 の三本柱。

### 1.2 初期数値表（チューニングのベースライン）

| パラメータ | 値 | 補足 |
| --- | --- | --- |
| 初期SP | **300** | ビルドフェーズの初期生産原資 |
| SP収入（本拠地） | **+5 / tick** | 占領中の本拠地ごと |
| SP収入（補給ノード） | **+3 / tick** | 占領中の補給ノードごと |
| CP回復 | **+10 / tick**（上限 **100**） | 0tick時点の上限 |
| CP上限の漸増 | **90tick以降 +2/tick ずつ上昇**（最大 **160**） | 終盤を加速させ膠着を防ぐ |
| 1tick最大行動数 | **6** | サーバー強制 |
| ラウンド制限 | **120 / 180 / 240 tick**（S/M/L） | 時間切れ時は占領ノード数で判定 |

### 1.3 ユニット三すくみ（コスト ⇔ 役割 ⇔ カウンター）

戦闘は3種のユニットのジャンケン（三すくみ）で回す。生産（SP消費）で揃え、相手の構成を読んでカウンターを当てる。

```
   突撃 ──（強い）──▶ 機動
     ▲                  │
（強い）             （強い）
     │                  ▼
   防衛 ◀──（強い）──── （防衛へ）
```

- **突撃 (assault)**: 機動に強い / 防衛に弱い。高攻撃・**対拠点◎（攻城）**＝要塞化を無効化して崩す。
- **防衛 (guard)**: 突撃に強い / 機動に弱い。高耐久・守備◎。
- **機動 (skirmish)**: 防衛に強い / 突撃に弱い。移動が速い・**索敵◎**（視界拡張）。

#### ユニット性能・コスト表

| ユニット | 生産SP | 稼働CP（移動/攻撃時） | 攻撃力 | HP | 移動 | 特性 |
| --- | --- | --- | --- | --- | --- | --- |
| 突撃 (assault) | 40 | 1 | 10 | 20 | 1/tick | 拠点防御を無視（攻城 ×2） |
| 防衛 (guard) | 50 | 1 | 6 | 35 | 1/tick | 守備時の被ダメ軽減が大きい |
| 機動 (skirmish) | 45 | 1 | 7 | 15 | 2/tick | 視界+1（索敵）。奇襲・迂回向き |

- **相性倍率**: 有利 **×1.5** / 不利 **×0.67**（例: 突撃が機動を攻撃すると攻撃力 ×1.5）。
- **拠点強化（fortify）**: 30SP/段で防御+8（最大Lv3）。守備側の実効HPと被ダメ軽減に寄与。要塞ノードはLv2、本拠地はLv1から開始。**突撃の攻城特性は要塞化分を無視** するため、タートル一辺倒は突撃で崩れる（=第4の軸）。

#### 戦闘解決の例（毎tick）

あるノードで攻撃側（突撃×3）が、守備側（機動×2、fortify Lv1）を攻める場合:

```
攻撃側実効攻撃 = 3 × 10(突撃atk) × 1.5(対機動・有利) = 45
  （突撃は攻城特性によりノードのfortify防御を無視）
守備側被ダメ = 45 → 機動HP15×2=30 を上回り守備全滅 → ノード占領

守備側実効攻撃 = 2 × 7(機動atk) × 0.67(対突撃・不利) ≒ 9.4
攻撃側被ダメ = 9.4 → 突撃HP20×3=60 のうち約9.4を消耗（1体未満の損耗）
→ 攻撃側はほぼ無傷で占領（相性を取った好例）
```

逆に守備が防衛ユニットなら突撃は不利（×0.67）になり、攻城特性があっても損耗が増える。**相性と拠点強化の読み合い** が戦闘の肝。

### 1.4 トポロジー設計が効く理由

- ユニットは **エッジで繋がった隣接ノードにしか移動できない**。経路設計＝どこに前線を作り、どこを補給線にするか。
- 補給ノードを多く押さえるほどSP収入が増えるが、占領面が広がると守る箇所も増える。
- 「本拠地直結の短い防衛線（守りやすいがSPが伸びない）」vs「補給ノードへ手を伸ばす拡張（稼げるが手薄になる）」のトレードオフが設計の核心。

### 1.5 「白熱」を生む3つの仕組み

1. **CP上限による同時行動制限**
   CPに上限があるため、AIは全ノードを同時にフル稼働できない。攻めれば守りのCPが、守れば攻めのCPが足りなくなる。常に「今このtick、何を優先するか」の取捨選択が発生する。

2. **ノード距離・レーン構造で多層防御に意味**
   本拠地に近いノードほど増援が届きやすく守りやすい。攻め込むほど補給線が伸びて手薄になる。深く攻めることと安全を保つことがトレードオフになり、一気押しが成立しにくい。

3. **攻撃/防御の相互リソース報酬で一方的展開を緩和**
   ノード奪取は奪った側にSP源を与えるが、撃退した側にもCP/SP報酬（防御成功ボーナス）が入る。攻め込まれている側も、守り切るたびに反撃原資が貯まる。劣勢側に自然なカムバック原資が供給され、勝者総取りの雪だるま式崩壊を防ぐ。

## 2. AI作戦ポリシー（IF-THEN）

### 2.1 基本構造

戦闘フェーズで自軍AIを動かすのは、プレイヤーがビルドフェーズで組んだ **優先順位付きルールリスト** である。

- ルールは **上から順に評価** され、条件を満たした最初のルールの行動を実行する。
- 条件・行動はそれぞれ **カード化** され、ドラッグ＆ドロップで組み立てる。
- **ルール枠には上限がある**（標準: 12枠）。攻め・守り・生産・索敵を全部は積めず、「何を捨てるか」を競う。

```
ルールリスト（上から評価）
┌────┬──────────────────────────────┬───────────────────────────┐
│ #  │ IF（条件カード）               │ THEN（行動カード）          │
├────┼──────────────────────────────┼───────────────────────────┤
│ 1  │ 本拠地HP < 30%                │ 全戦力で本拠地防衛          │
│ 2  │ 敵が本拠地に隣接              │ 防衛集中 ＋ 本拠地を要塞化   │
│ …  │ …                            │ …                          │
│ 12 │ （デフォルト）                │ 最寄りの中立ノードを占領     │
└────┴──────────────────────────────┴───────────────────────────┘
```

### 2.2 条件カテゴリ（IF）

| カテゴリ | 例となる条件カード |
| --- | --- |
| 占領状況 | `自軍ノード数 < X`, `占領率 < X%`, `敵ノード数 > 自軍` |
| 本拠地 | `本拠地HP < X%`, `敵が本拠地に隣接` |
| ノード | `任意ノード防御 < X`, `補給ノードが中立` |
| 隣接戦力 | `隣接の敵戦力 = 少/中/多` |
| 自軍CP | `自CP < X`, `余剰CP > X%` |
| 自軍SP | `自SP > X`, `自SP < X` |
| ユニット構成 | `敵の突撃が多い`, `自軍に防衛が居ない` |
| 経過時間 | `経過 > X tick`, `残り時間 < X tick` |
| イベント | `ノード奪取`, `ノード喪失`, `同一ノードN回奪還`, `逆転ゲージ満タン` |

### 2.3 行動カテゴリ（THEN）

| カテゴリ | 例となる行動カード |
| --- | --- |
| 進軍 | `隣接ノードXへ進軍`, `敵本拠地へ直行`, `最弱の隣接ノードを攻撃`, `要塞を回避して迂回` |
| 防御 | `指定ノードへ増援`, `本拠地防衛集中`, `拠点強化（fortify）` |
| 生産 | `突撃を生産`, `防衛を生産`, `機動を生産`（敵構成にカウンター） |
| 索敵 | `機動で偵察`, `手薄レーンへ戦力集中` |
| 攪乱 | `陽動でデコイ前進` |
| 撤退/温存 | `劣勢ノードから後退`, `攻撃停止しCP温存` |
| スペシャル | `増援サージ発動`, `強襲発動`, `焦土作戦発動` |

### 2.4 ルール具体例（12例）

> 即運用可能なルール。上ほど優先度が高い想定。

| # | IF（条件） | THEN（行動） | 意図 |
| --- | --- | --- | --- |
| 1 | 本拠地HP < 30% | 全戦力で本拠地防衛 | 崩壊回避を最優先 |
| 2 | 敵が本拠地に隣接 | 防衛集中＋本拠地を要塞化 | 最終防衛線の構築 |
| 3 | 隣接ノードの敵戦力 = 少 | 進軍してそのノードを奪取 | 弱点を突いて占領拡大 |
| 4 | 自CP < 20% | 進軍停止しCP回復を待つ | 手詰まり防止・温存 |
| 5 | 敵ユニットに突撃が多い | 防衛ユニットを優先生産 | カウンター生産 |
| 6 | 同一ノードを3回奪還成功 | そのノードを要塞化しチョーク化 | 成功パターンの固定化 |
| 7 | 占領率が優勢 かつ 残り時間 < 60tick | 防衛全振りで時間切れ勝ちを狙う | リードの保全 |
| 8 | 占領率が劣勢 | 補給ノードへ進軍しSP収入を奪取 | 経済の立て直し |
| 9 | 機動の偵察で敵手薄レーンを発見 | そのレーンへ戦力集中 | 情報を攻めに転換 |
| 10 | 余剰SP > 200 | 前線ノードを拠点強化 | 余力を防御に投資 |
| 11 | 逆転ゲージ満タン かつ 劣勢 | 増援サージを発動 | 逆転札の切り時 |
| 12 | （デフォルト/上記いずれも不成立） | 最寄りの中立ノードを占領 | SP源の基本拡大 |

### 2.5 ルール枠の戦略的意味

12枠の上限により、プレイヤーは「思想」の取捨選択を迫られる。

- **速攻型**: 進軍・生産ルールを厚くし、敵本拠地直行で即時勝利を狙う。
- **経済型**: 補給ノード確保・拠点強化を軸に、収入差で押し切る。
- **守勢型**: 防衛・要塞化を厚くし、時間切れの占領数判定を狙う。

枠は有限なので「全部入り」はできず、相手の型を読んでメタを張る読み合いが生まれる。これがビルドフェーズの主戦場である。

## 3. 逆転要素（スペシャル）

### 3.1 設計方針

- すべてのスペシャルは **溜め（ゲージ）＋明確な発動条件** を持ち、運要素は一切排除する。
- 効果は **強力だが必ずリスク（デバフ・自軍ペナルティ）を伴う**。
- 逆転札の発動が成立したときの **最終勝率が概ね50%前後** に収まるよう、発動条件を意図的に重くする。

### 3.2 スペシャル一覧

| スペシャル | 溜め条件 | 発動条件 | 効果 | リスク／コスト |
| --- | --- | --- | --- | --- |
| **増援サージ** | 「劣勢ゲージ」（敵占領優位・ノード喪失で増加） | ゲージ満タンで発動 | 一定時間（10tick）SP収入 ×2 ＋ 全ユニット攻撃 +30% | 使用後、生産を5tick停止 |
| **強襲（戦況把握）** | 「偵察ゲージ」（索敵・時間で蓄積） | ゲージ100%で **一度だけ** | 全マップ可視化（15tick）＋ 敵の増援を4tick停止 ＋ 敵CPドレイン(-40) | 発動後ゲージリセット・再充填は遅い |
| **焦土作戦** | 常時待機（条件のみ） | 本拠地HP < 10% でのみ発動可 | 占領されかけた自陣ノードを中立化リセット ＋ 本拠地を要塞化(+2) | 自軍も罰: SP収入が残ラウンド半減 |

### 3.3 劣勢ゲージの設計（増援サージ用）

```
劣勢ゲージ増加 = base
  + (敵ノード数 - 自ノード数) × 係数a     // 占領で負けるほど増
  + (このtickに失ったノード数) × 係数b     // ノードを失うほど増
  - (自軍が優勢な時間) × 係数c            // 優勢だと減衰

→ 圧倒的劣勢でのみ満タンになる設計。優勢側は基本撃てない。
```

### 3.4 勝率50%設計の考え方

スペシャルは「劣勢を五分に戻す」ためのものであり、「優勢をさらに広げる」ものにしない。増援サージは劣勢でしか溜まらず、強襲は重い偵察前提、焦土は自軍も罰を受ける——いずれも軽率には撃てない。結果として逆転札が決まっても「そこから改めて五分の勝負」になり、撃った側の **その後の作戦ポリシーの質** が最終的な勝敗を決める。運ではなく設計と判断で逆転する。

## 4. AI連携 3レイヤー概要

本作の作戦AIは、プレイヤーの習熟度・目的に応じて3つの実装レイヤーから選べる。**いずれもサーバー権威・決定論的判定** の上で動く（詳細は `ai-interface.md`）。

| レイヤー | 名称 | 実行環境 | 想定ユーザー | 概要 |
| --- | --- | --- | --- | --- |
| **L1** | 組み込みルールエンジン | サーバー内蔵 | 全プレイヤー（入門〜上級） | 第2章のIF-THENを内蔵ビヘイビアツリーとして評価。最も公平。**デモのNPCはこれ** |
| **L2** | 簡易DSL / ノードグラフ | サーバー側サンドボックス | 中〜上級・自作派 | 軽量DSLでL1より複雑な分岐・状態保持を記述（デモではスコープ外＝拡張ポイント） |
| **L3** | 自前LLM / 外部AIエンドポイント | プレイヤー側 + サーバー検証 | 上級・研究/開発者（**本企画の主役**） | プレイヤーのLLM（ローカル/外部）を作戦AIとして接続。**デモで自前AIとして接続可能**。詳細は `ai-interface.md` |

### レイヤー間の関係

- L1/L2/L3は **同一ルール・同一サーバー判定** の上で対戦可能（混在マッチも設計上可能）。
- どのレイヤーでも、行動はサーバーが検証・適用する。CP超過・不正行動・見えない敵への攻撃は一律拒否され、勝敗判定はサーバーが権威を持つ。
- 本リポジトリの `demo/` は **L1（NPC）対 L3/自前JSポリシー（あなたのAI）** を実際に動かせる実証環境である。
